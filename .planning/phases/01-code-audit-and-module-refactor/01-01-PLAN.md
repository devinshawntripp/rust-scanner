    Per CONTEXT.md, there are 47 unwrap/expect calls across 11 files. The hotspots are: archive.rs (15), redhat.rs (9), main.rs (6), binary.rs (4), vuln/tests.rs (4).

    **Assessment approach for each call:**
    1. Run `grep -n "\.unwrap()\|\.expect(" src/{file}` on each hotspot file
    2. For each call, determine: Can this panic on real user data? Or is it provably safe (e.g., unwrap on a just-checked Option, unwrap on a hardcoded regex, unwrap_or_default)?
    3. Categorize: SAFE (leave), DANGEROUS (fix), TEST-ONLY (leave)

    **Fix DANGEROUS unwraps only — per user decision, leave safe patterns:**
    - Convert dangerous `.unwrap()` to `.unwrap_or_default()`, `.ok()?`, `match`, or `.context("...")?` using anyhow
    - archive.rs: Focus on file I/O unwraps that can fail on malformed archives
    - redhat.rs: Focus on XML parsing unwraps that can fail on unexpected OVAL data
    - binary.rs: Focus on goblin parsing unwraps that can fail on corrupted binaries
    - main.rs: Most are likely CLI arg unwraps which are safe (clap validates), but check

    **Do NOT touch:**
    - `unwrap()` in test code (vuln/tests.rs) — tests should panic on failure
    - `unwrap_or_default()` patterns — already safe
    - `unwrap()` on hardcoded values like `Regex::new("known-valid-pattern").unwrap()` — provably safe
    - `expect("message")` where the expect message documents an invariant that truly cannot fail

    **Also look for and fix:**
    - Regex patterns compiled inside loops — move to OnceLock statics (per CONTEXT.md, existing pattern used for HTTP clients)
    - Only do this where the same regex is compiled multiple times in hot paths
